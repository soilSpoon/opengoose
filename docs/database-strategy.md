> **대상 프로젝트:** OpenGoose
> **문서 유형:** 의사결정 문서 (ADR)

## 1. 현재 아키텍처 분석

### 1.1 SQLite + Diesel ORM 기반 스키마

OpenGoose는 현재 SQLite를 Diesel ORM(Rust)으로 감싸서 사용하며, 단일 임베디드 바이너리로 배포된다. 전체 13개 테이블이 4개의 도메인으로 분류된다.

**채널/채팅 이력 (3개 테이블)**

| 테이블 | 컬럼 수 | 역할 |
|--------|---------|------|
| `sessions` | 5 | 세션 관리 (`session_key`, `active_team`, 타임스탬프) |
| `messages` | 6 | 채팅 메시지 저장 (`role`, `content`, `author`) |
| `message_queue` | 13 | 에이전트 간 메시지 큐 (`sender`→`recipient`, `status`, 재시도 로직) |

**작업 추적/오케스트레이션 (2개 테이블)**

| 테이블 | 컬럼 수 | 역할 |
|--------|---------|------|
| `work_items` | 14 | 계층적 작업 항목 (`parent_id`로 트리 구조), 할당/상태 추적 |
| `orchestration_runs` | 12 | 워크플로우 실행 추적 (`current_step`/`total_steps` 진행률) |

**에이전트 간 통신 (2개 테이블)**

| 테이블 | 컬럼 수 | 역할 |
|--------|---------|------|
| `agent_messages` | 9 | 에이전트 간 직접 통신 (상태: Pending→Delivered→Acknowledged) |
| `event_history` | 6 | 감사 추적 (`event_kind`, `source_gateway`, JSON `payload`) |

**모니터링/자동화/시스템 구성 (6개 테이블)**

| 테이블 | 컬럼 수 | 역할 |
|--------|---------|------|
| `alert_rules` | 10 | 알림 규칙 정의 (`metric`, `condition`, `threshold`) |
| `alert_history` | 6 | 알림 발생 이력 |
| `schedules` | 10 | 크론 기반 스케줄링 |
| `triggers` | 11 | 조건부 트리거 (`condition_json`) |
| `plugins` | 10 | 플러그인 레지스트리 |
| `api_keys` | 5 | API 키 관리 (`key_hash`) |

### 1.2 WAL 모드의 장점과 한계

현재 SQLite는 WAL(Write-Ahead Logging) 모드로 운영된다.

**장점:**
- 읽기와 쓰기가 서로를 차단하지 않아 동시 읽기 성능이 우수
- 단일 바이너리 배포 — 별도 DB 서버 운영 불필요
- 제로 설정, 제로 의존성
- 로컬 개발 환경에서 즉시 사용 가능

**한계:**
- **단일 쓰기자 제약**: 동시에 하나의 프로세스만 쓰기 가능. 멀티에이전트가 병렬로 쓰기를 시도하면 `SQLITE_BUSY` 발생
- **네트워크 접근 불가**: 파일 시스템 기반이므로 원격 에이전트가 직접 접근 불가
- **스케일링 천장**: 프로세스 내 단일 커넥션 풀로 제한

### 1.3 Diesel ORM 투자 비용과 마이그레이션 영향

Diesel ORM에 대한 현재 투자는 상당하다:

- **13개 테이블 스키마 정의** (`schema.rs` 197줄)
- **타입 안전 쿼리**: 컴파일 타임에 SQL 오류를 잡아내는 Rust 매크로 기반
- **JOIN 관계**: `allow_tables_to_appear_in_same_query!`로 크로스 테이블 쿼리 지원
- **테스트 커버리지**: 컬럼 존재 확인, nullable 검증, JOIN 컴파일 테스트 등 포괄적 테스트

마이그레이션 비용 추정:

| 대상 DB | Diesel 지원 | 마이그레이션 난이도 | 비고 |
|---------|-------------|-------------------|------|
| PostgreSQL | `diesel(postgres)` 피처 플래그 | **중간** | Diesel이 공식 지원. 타입 매핑 변경 필요 (`Text`→`Varchar`, `Integer`→`Int4` 등) |
| Dolt (MySQL 프로토콜) | `diesel(mysql)` 피처 플래그 | **중간~높음** | MySQL 와이어 프로토콜 호환이지만, Dolt 고유 기능(branching)은 raw SQL 필요 |
| SurrealDB | 미지원 | **매우 높음** | ORM 전체 교체 필요 |

## 2. 멀티에이전트 오케스트레이션의 데이터 요구사항

### 2.1 OpenGoose 에코시스템의 실제 요구사항

OpenGoose가 지원하는/지원할 세 가지 운용 모델에서 데이터베이스에 요구하는 것은 서로 다르다.

**Gas Town (중앙 집중형 멀티에이전트)**
- 20-30개 에이전트가 동시에 작업 항목을 읽고/쓰기
- `work_items` 테이블에 대한 높은 동시 쓰기 부하
- `message_queue`와 `agent_messages`의 실시간 상태 전환
- 실사용 중: 단일 Dolt 서버(포트 3307)에서 4개 DB(`hq`, `beads`, `gastown`, `crew`) 운영

**Wasteland (분산/연합)**
- 지리적으로 분산된 에이전트 노드
- 네트워크 파티션 내성 필요
- 데이터 동기화 및 충돌 해결 메커니즘 필수

**Goosetown (로컬/소규모)**
- 2-5개 에이전트의 로컬 오케스트레이션
- 현재 SQLite로 충분히 운영 가능
- 설정 단순성이 최우선

### 2.2 버전 관리(브랜칭/머징) — 실제로 필요한 경우

데이터 브랜칭이 단순한 "있으면 좋은 기능"이 아니라 **필수인 시나리오**:

| 시나리오 | 브랜칭 필요성 | 설명 |
|---------|-------------|------|
| 에이전트 실험적 작업 | **높음** | 에이전트가 작업을 시도하고, 실패하면 롤백해야 할 때 |
| 병렬 작업 계획 수립 | **중간** | 여러 에이전트가 동일 작업에 대해 다른 접근법을 시도 |
| 감사/규정 준수 | **낮음** | temporal tables로 충분 |
| 일상적 CRUD | **없음** | 대부분의 읽기/쓰기 작업 |

**결론**: 브랜칭은 "탐색적 에이전트 작업"에서만 진정한 가치가 있다. 대부분의 오케스트레이션 시나리오에서는 선형 이력(temporal tables)이면 충분하다.

### 2.3 실시간 상태 추적

`agent_messages`의 상태 전환(Pending→Delivered→Acknowledged)과 `work_items`의 상태 변경은 실시간으로 관찰 가능해야 한다. 현재 SQLite에서는 폴링으로 구현해야 하며, 이는 에이전트 수가 증가하면 비효율적이다.

**요구 수준별 솔루션:**
- **폴링 (현재)**: SQLite — 에이전트 5개 이하에서 작동
- **DB 레벨 알림**: PostgreSQL LISTEN/NOTIFY — 중간 규모
- **전용 메시징**: Redis Pub/Sub — 대규모, 서브밀리초 지연

### 2.4 감사 추적과 책임 소재

`event_history` 테이블이 현재 이 역할을 하지만, 다음이 부족하다:
- 변경 "이전" 상태 기록 없음 (현재는 이벤트만 기록)
- 변경을 수행한 에이전트의 의사결정 맥락 미보존
- 시간 기반 쿼리("3시간 전 상태는?") 어려움

## 3. 옵션별 상세 평가

### 3.1 Dolt (Git-for-Data)

**개요**: MySQL 와이어 프로토콜을 사용하는 "Git for Data" SQL 데이터베이스. Prolly Trees 기반의 셀 레벨 3-way 머지를 네이티브로 지원한다.

**강점:**

- **네이티브 브랜칭/머징**: 에이전트가 데이터 브랜치를 생성하고, 실험하고, 머지하거나 폐기할 수 있음
- **셀 레벨 3-way 머지**: 두 에이전트가 같은 테이블의 다른 행을 수정하면 자동 머지
- **MySQL 호환**: 기존 MySQL 클라이언트/ORM 그대로 사용 가능
- **MCP 서버 제공**: AI 에이전트가 직접 DB를 조작할 수 있는 인터페이스
- **호스팅 옵션**: AWS/GCP에서 Hosted Dolt 사용 가능
- **쓰기 성능**: MySQL 대비 10% 빠른 쓰기

**약점:**

- **읽기 성능**: MySQL 대비 ~30% 느림
- **TPC-C 처리량**: MySQL의 40% 수준 — 높은 트랜잭션 부하에서 병목 가능
- **실시간 Pub/Sub 없음**: 상태 변경 알림을 위해 폴링 또는 별도 메시징 시스템 필요
- **Rust 에코시스템**: `diesel(mysql)` 피처로 사용 가능하나, Dolt 고유 기능(DOLT_BRANCH, DOLT_MERGE 등)은 raw SQL로 호출해야 함

**Gas Town 실사용 경험:**

Gas Town은 현재 Dolt를 프로덕션에서 운영 중이다:
- 단일 Dolt 서버, 포트 3307
- 4개 데이터베이스: `hq`(본부), `beads`(화폐/보상), `gastown`(인프라), `crew`(인원)
- Python에서 DoltPy v2.0.14 + pymysql/mysql-connector로 접근
- 브랜칭은 주로 에이전트의 탐색적 작업에 활용

이 실사용 경험은 Dolt가 멀티에이전트 환경에서 **작동한다**는 것을 검증하지만, Gas Town 규모(20-30 에이전트)에서의 읽기 성능 문제가 보고되었는지는 추가 확인이 필요하다.

**OpenGoose 적합성 평가: ★★★★☆**

브랜칭이 필요한 시나리오에서는 유일한 선택지. 그러나 읽기 성능 저하와 실시간 알림 부재는 보완이 필요하다.

### 3.2 PostgreSQL + Temporal Tables

**개요**: 세계에서 가장 성숙한 오픈소스 RDBMS에 시간 기반 이력 추적을 추가하는 패턴.

**강점:**

- **입증된 안정성**: 30년 이상의 프로덕션 검증. 장애 대응 노하우가 풍부
- **LISTEN/NOTIFY**: 데이터베이스 레벨의 실시간 변경 알림. `agent_messages` 상태 전환을 즉시 감지 가능
- **Temporal Tables**: 확장 또는 트리거 기반으로 모든 변경의 이전/이후 상태를 자동 기록
- **Diesel 공식 지원**: `diesel(postgres)` 피처 플래그로 전환. 스키마 매핑이 가장 쉬움
- **에코시스템**: Grafana, Metabase 등 대시보드 도구와의 연동이 뛰어남
- **동시 쓰기**: MVCC 기반으로 멀티에이전트 병렬 쓰기에 제약 없음
- **Rust 지원**: Diesel 외에도 `sqlx`, `tokio-postgres` 등 다양한 비동기 드라이버 존재

**약점:**

- **브랜칭/머징 없음**: 선형 이력만 제공. 에이전트가 "이 변경을 되돌리고 싶다"면 수동으로 역변환 필요
- **운영 부담**: 별도 서버 프로세스, 백업, 모니터링 필요
- **임베디드 불가**: SQLite처럼 단일 바이너리에 포함할 수 없음

**Diesel 마이그레이션 경로:**

```
# Cargo.toml 변경
diesel = { features = ["postgres"] }  # "sqlite" 대신

# 필요한 타입 변경 예시
# Text → Varchar (대부분 호환)
# Integer → Int4 (대부분 호환)
# Double → Float8
# Nullable<T> → Nullable<T> (변경 없음)
```

현재 OpenGoose 스키마의 13개 테이블은 모두 표준 SQL 타입만 사용하므로, PostgreSQL로의 마이그레이션은 기계적으로 수행 가능하다. 가장 큰 작업은 `Text` 타임스탬프를 PostgreSQL의 네이티브 `Timestamp` 타입으로 전환하는 것이다.

**OpenGoose 적합성 평가: ★★★★☆**

브랜칭이 불필요한 대부분의 시나리오에서 최적의 선택. 특히 LISTEN/NOTIFY는 현재 OpenGoose에 누락된 실시간 상태 추적을 가장 낮은 비용으로 해결한다.

### 3.3 SurrealDB

**개요**: 문서/그래프/관계형을 모두 지원하는 멀티모델 DB. v3.0 GA.

**강점:**
- WebSocket 기반 실시간 라이브 쿼리
- 내장 인증 및 멀티테넌시
- 유연한 데이터 모델링

**약점:**
- **미성숙한 에코시스템**: 프로덕션 검증 사례 부족
- **SurrealQL ≠ SQL**: 기존 Diesel ORM 전체 교체 필요
- **브랜칭 미지원**
- **성능 우려**: 대규모 환경에서의 벤치마크 데이터 부족
- **Diesel 미지원**: ORM 레이어를 완전히 재작성해야 함

**결론**: 흥미로운 기술이지만, OpenGoose가 감수할 리스크 대비 이점이 불충분하다. 현재 단계에서는 **선택지에서 제외**한다. 1-2년 후 에코시스템 성숙도를 재평가할 수 있다.

**OpenGoose 적합성 평가: ★★☆☆☆**

---

### 3.4 Redis + PostgreSQL (하이브리드)

**개요**: Redis를 실시간 메시징/캐싱 계층으로, PostgreSQL을 영구 저장소로 사용하는 업계 표준 패턴.

**강점:**

- **업계 표준**: CrewAI, LangGraph, AutoGen 등 주요 AI 에이전트 프레임워크가 채택
- **Redis 채택률**: AI 에이전트 데이터 저장소 분야 1위 (43% 점유율)
- **서브밀리초 지연**: `agent_messages` 상태 전환, `message_queue` 처리에 이상적
- **Redis Streams**: `message_queue` 테이블을 완전히 대체 가능. 소비자 그룹으로 에이전트별 큐 분리
- **Redis Pub/Sub**: 실시간 이벤트 브로드캐스트
- **PostgreSQL 보완**: 영구 저장이 필요한 데이터(`work_items`, `event_history`, `sessions`)는 PostgreSQL에 보관

**약점:**

- **두 시스템 운영**: Redis + PostgreSQL 모두 관리해야 함. 운영 복잡도 2배
- **데이터 일관성**: Redis와 PostgreSQL 간 동기화 문제. 장애 시 데이터 유실 가능
- **브랜칭 없음**: 두 시스템 모두 네이티브 브랜칭 미지원
- **과잉 설계 위험**: OpenGoose의 현재 규모에서는 불필요한 복잡성

**적합한 경우:**
- 에이전트 50개 이상의 대규모 배포
- 서브밀리초 메시지 전달이 비즈니스 요구사항인 경우
- 이미 Redis/PostgreSQL 인프라가 존재하는 경우

**OpenGoose 적합성 평가: ★★★☆☆**

기술적으로 우수하지만, 현재 OpenGoose 규모에서는 복잡도 대비 이점이 부족하다.

### 3.5 CRDTs / SQLite + Litestream

**CRDTs (Automerge 등)**

- 충돌 없는 동시 수정이 가능하지만, 이것은 **데이터베이스가 아니다**
- SQL 없음, 인덱스 없음, 쿼리 언어 없음
- Python 지원 제한적, Rust 지원은 존재하나 프로덕션 사례 부족
- 공동 편집(collaborative editing)에는 적합하나, `work_items`나 `orchestration_runs` 같은 구조화된 데이터 관리에는 부적합

**결론**: OpenGoose의 요구사항과 근본적으로 맞지 않는다.

**SQLite + Litestream**

- Litestream은 SQLite WAL을 S3에 스트리밍하여 복제를 제공
- **치명적 한계**: 단일 쓰기자 제약이 그대로 유지됨. 멀티에이전트 병렬 쓰기 불가
- Fly.io가 에이전트 지향 기능을 개발 중이나, 현재로서는 미래의 약속일 뿐
- 단일 에이전트 시나리오에서만 유효

**결론**: 멀티에이전트 오케스트레이션이 목표인 OpenGoose에는 **구조적으로 부적합**.

**OpenGoose 적합성 평가: ★☆☆☆☆**

## 5. OpenGoose 시나리오별 분석

### 시나리오 A: 현재 규모 유지 (2-5 에이전트)

**권장: SQLite 유지**

- WAL 모드에서 2-5개 에이전트의 순차적 쓰기는 충분히 처리 가능
- `message_queue`의 폴링 기반 상태 확인도 이 규모에서는 성능 문제 없음
- 제로 운영 부담은 개발 속도에 직접적 이점
- Diesel ORM 투자를 그대로 활용

**전환 신호**: 다음 중 하나라도 해당되면 마이그레이션 검토 시작
- `SQLITE_BUSY` 오류가 분당 10회 이상 발생
- 에이전트가 6개 이상으로 확장
- 원격 에이전트 접근이 필요해질 때

### 시나리오 B: Gas Town 수준 (20-30 에이전트)

**권장: Dolt 우선 검토, 대안으로 PostgreSQL**

이 규모에서 SQLite의 단일 쓰기자 제약은 실질적 병목이 된다.

**Dolt를 선택하는 경우:**
- 에이전트의 탐색적 작업이 핵심 워크플로우에 포함될 때
- "에이전트가 데이터 변경을 시도하고 결과를 검증한 후 머지"하는 패턴이 빈번할 때
- Gas Town과의 기술 스택 통일이 가치 있을 때

**PostgreSQL을 선택하는 경우:**
- 브랜칭보다 읽기 성능과 실시간 알림이 더 중요할 때
- Diesel의 PostgreSQL 지원이 더 안정적이고 마이그레이션 비용이 낮을 때
- 대시보드/모니터링 인프라(Grafana 등)가 필요할 때

### 시나리오 C: 연합/분산 (Wasteland)

**권장: PostgreSQL 필수, Dolt는 특수 계층으로 추가 검토**

분산 환경에서는 네트워크를 통한 데이터 접근이 기본 요구사항이므로 임베디드 DB는 배제된다.

**기본 구성:**
- PostgreSQL을 각 노드의 기본 저장소로 사용
- 노드 간 동기화는 PostgreSQL의 Logical Replication 또는 별도 동기화 프로토콜

**Dolt 추가 고려:**
- 노드 간 데이터를 "pull/push"하는 Git 스타일 동기화가 필요하면 Dolt의 리모트 기능 활용 가능
- 단, Dolt의 읽기 성능(MySQL 대비 -30%)이 분산 환경에서 누적되면 심각한 병목 가능

## 6. 권장 경로

### 6.1 현재: SQLite 유지 (변경 없음)

OpenGoose의 현재 상태에서 데이터베이스 마이그레이션은 **시기상조**다.

이유:
1. 13개 테이블, 단순 타입 — SQLite가 완벽히 처리하는 범위
2. Diesel ORM 투자(스키마 정의, 타입 안전 쿼리, 테스트)를 보존
3. 제로 운영 부담으로 기능 개발에 집중 가능
4. 단일 바이너리 배포의 사용자 경험 이점

**지금 해야 할 일**: 마이그레이션 준비를 위한 추상화 계층 도입
- 데이터 접근 코드를 trait 기반으로 추상화하여 DB 교체 시 영향 범위 최소화
- 타임스탬프를 `Text`에서 구조화된 타입으로 점진적 전환 (PostgreSQL 마이그레이션 대비)

### 6.2 조건부 마이그레이션: 언제, 왜, 어떻게

**마이그레이션 트리거 조건 (하나 이상 충족 시):**

| 조건 | 대상 DB | 근거 |
|------|---------|------|
| 에이전트 10개 이상, 동시 쓰기 병목 발생 | PostgreSQL | 동시 쓰기 해결이 최우선 |
| 에이전트 탐색적 브랜칭이 핵심 기능으로 확정 | Dolt | 유일한 네이티브 브랜칭 제공자 |
| 실시간 상태 추적이 비즈니스 요구사항 | PostgreSQL | LISTEN/NOTIFY가 가장 낮은 복잡도 솔루션 |
| 원격/분산 에이전트 지원 필요 | PostgreSQL 또는 Dolt | 네트워크 접근 가능한 DB 필수 |
| 시간 기반 이력 조회 필요 ("1시간 전 상태") | Dolt | `AS OF` 쿼리가 가장 자연스러움 |

**최우선 마이그레이션 대상: PostgreSQL**

대부분의 전환 시나리오에서 PostgreSQL이 최선이다:
- Diesel 마이그레이션 비용이 가장 낮음
- 동시 쓰기, 실시간 알림, 이력 추적을 모두 해결
- 운영 노하우와 에코시스템이 가장 풍부

**Dolt를 선택해야 하는 유일한 경우**: 데이터 브랜칭/머징이 OpenGoose의 차별화 기능으로 확정될 때. 이 경우에도 Dolt의 읽기 성능 한계를 감안하여 읽기 캐시(Redis 등) 추가를 계획해야 한다.

### 6.3 마이그레이션 단계 (Dual-Write 전략)

데이터베이스 전환은 빅뱅이 아니라 점진적으로 수행한다.

```
단계 1: Dual-Write (2-4주)
┌─────────┐     ┌──────────┐
│ OpenGoose├──┬──▶│  SQLite   │  ← 기존 (Primary Read + Write)
│ 애플리케이션 │  │  └──────────┘
│         │  │  ┌──────────┐
│         │  └──▶│ 새 DB     │  ← Shadow Write (검증용)
└─────────┘     └──────────┘
- 모든 쓰기를 양쪽에 수행
- 읽기는 SQLite에서만
- 양쪽 데이터 일치 검증

단계 2: Validate (1-2주)
- Shadow DB의 데이터 무결성 검증
- 쿼리 성능 벤치마크 (특히 JOIN 쿼리)
- 에이전트 통합 테스트

단계 3: Switch Reads (1-2주)
┌─────────┐     ┌──────────┐
│ OpenGoose├──┬──▶│  SQLite   │  ← Write 유지 (폴백용)
│ 애플리케이션 │  │  └──────────┘
│         │  │  ┌──────────┐
│  (읽기) ├──┴──▶│ 새 DB     │  ← Primary Read + Write
└─────────┘     └──────────┘
- 읽기를 새 DB로 전환
- SQLite는 쓰기만 계속 (롤백 대비)
- 오류 발생 시 즉시 SQLite로 롤백

단계 4: Drop Old (확인 후)
- SQLite 쓰기 중단
- SQLite 파일은 백업으로 보관 (삭제하지 않음)
- 마이그레이션 완료
```

**핵심 원칙:**
- 각 단계에서 롤백이 가능해야 한다
- 데이터 검증 없이 다음 단계로 진행하지 않는다
- 마이그레이션 중 서비스 중단은 허용하지 않는다
